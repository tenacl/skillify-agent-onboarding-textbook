Chapter 1. 에이전트의 기억 문제

🏛️ 여박총피법 如剝蔥皮

"양파를 한 겹씩 벗기듯, '왜?'를 한 단계 더 파고 들어간다."

에이전트는 왜 같은 실수를 반복할까?

대부분의 AI 에이전트는 세션 단위 기억만 가진다. 대화가 끝나면 컨텍스트 창이 사라지고, 다음 세션의 에이전트는 이전 실수를 전혀 모른다.

일반적인 "수정" 방식을 보자:

  • 프롬프트 미세 조정 — "Please don't hallucinate" 주문
  • 시스템 메시지 확대 — 점점 길어지는 지시문
  • 더 큰 모델 교체 — 비용만 올라가고 근본 구조는 동일

Garry Tan은 이걸 "vibes-based reliability"라고 표현했다. 분위기로 신뢰성을 확보하려는 것이다. 대화가 복잡해지는 순간 전부 무너진다.

왜 이게 필요한가? — 한 겹 더

$160M을 투자받은 LangChain도 테스트 도구(gym membership)는 줬지만 워크플로우(workout plan)는 빠뜨렸다. 무엇을 테스트할지, 어떤 순서로, 언제 끝인지 — 그 루프가 없었다. 개발자 대다수는 에이전트를 테스트하지 않는다. 프레임워크가 도구만 주고 방향을 안 줬기 때문이다.

🔍 비교로 이해하기 (피차비대법)
전통 소프트웨어AI 에이전트 (현재)
버그 발생테스트 작성 → 영구 방지프롬프트 수정 → 같은 버그 재발
기억코드베이스에 영구 저장세션 끝나면 소멸
품질 보증CI/CD, 자동 테스트"분위기"

반대로 생각하면?

만약 소프트웨어 팀에서 버그가 나올 때마다 테스트를 안 쓰고 "다음엔 조심하자"만 한다면? 2005년에 이미 해결한 문제를 에이전트 세계에서는 아직 방치하고 있다.

Chapter 2. Latent vs Deterministic

🏛️ 층체판석법 層遞判析

"단계별로 차곡차곡 분해하라. 한 번에 결론 내리지 말 것."

두 개의 공간

Garry Tan의 "thin harness, fat skills" 아키텍처에서 가장 핵심적인 구분:

Latent SpaceDeterministic Space
뭐가 필요?판단, 추론, 창의성정밀성, 일관성, 속도
누가 해야?LLM스크립트 (코드)
예시"이 이메일 분위기가 어때?"UTC→KST 시간 변환
실행 시간수 초수 밀리초
오류 특성때때로 틀림 (확률적)같은 입력 = 항상 같은 출력

실제 실패 사례 1: calendar-recall

에이전트에게 "10년 전 싱가포르 출장 언제였어?"라고 물었다. 로컬에 3,146개 캘린더 파일이 이미 인덱싱돼 있었다. grep이면 1밀리초 이내.

에이전트가 한 일:

  1. 라이브 캘린더 API 호출 → 너무 오래된 데이터라 차단
  2. 이메일 검색 → 노이즈 투성이
  3. API 파라미터 바꿔서 재시도 → 여전히 차단
  4. 5분 후 로컬 KB 검색 → 즉시 발견

버그의 정체: 틀린 답이 아니라 틀린 공간에서 작업한 것. 결정론적 작업을 latent space에서 처리했다.

실제 실패 사례 2: "28분"

에이전트: "다음 미팅 28분 후입니다." → 실제: 88분 후. UTC→PT 시간대 변환을 머릿속(latent)으로 해서 1시간 오차.

context-now.mjs라는 스크립트가 이미 존재했고, 50ms에 정확한 JSON을 반환한다. 에이전트가 안 썼을 뿐.

{
  "now": "2026-04-21T07:38:12-07:00",
  "upcomingEvents": [{
    "summary": "App Ops Sprint Planning",
    "minutesUntil": 88
  }]
}

The Loop — 핵심 메커니즘

Latent space가 결정론적 도구를 빌드한다 → 결정론적 도구가 latent space를 제약한다.

모델의 지능이 "모델이 멍청해지는 것을 막는 제약"을 직접 만든다. 에이전트가 calendar-recall.mjs를 작성하면, 이후 그 스킬은 에이전트에게 "라이브 API 대신 이 스크립트를 실행하라"고 강제한다. 구 실패 경로는 구조적으로 도달 불가능해진다.

Chapter 3. Skillify — 실패를 영구 구조로 바꾸기

🏛️ 조례최중법 條例最重

"규칙을 먼저 정한다. 기준 없이 시작하지 않는다."

핵심 개념

Skillify = 실패가 발생할 때마다 그것을 SKILL.md + 결정론적 코드 + 테스트로 변환하는 방법론. 10단계 체크리스트를 모두 통과해야 "skill"이다. 그렇지 않으면 "오늘만 작동하는 코드".

Garry Tan의 표현: "Every failure becomes a skill. Every skill has evals. Every eval runs daily."

Skillify 10단계 체크리스트

#단계핵심
1SKILL.md계약서 — 이름, 트리거, 규칙
2결정론적 코드scripts/*.mjs — 코드가 할 수 있는 건 LLM 금지
3Unit Tests순수 함수, 픽스처 데이터, vitest
4Integration Tests실제 데이터/엔드포인트
5LLM EvalsLLM-as-judge, 프로세스까지 평가
6Resolver TriggerAGENTS.md에 라우팅 진입점 등록
7Resolver Eval트리거가 실제로 라우팅되는지 검증
8Check-resolvable + DRY도달 불가 스킬, 중복 제거
9E2E Smoke Test전체 파이프라인 end-to-end
10Brain Filing RulesKB 저장 위치 규칙

왜 이게 필요한가?

테스트 없는 스킬 시스템의 실패 모드 3가지:

  1. 중복 스킬 — 월요일에 deploy-k8s, 목요일에 kubernetes-deploy. 같은 트리거, 다른 스킬. 라우팅이 주사위가 됨.
  2. 사일런트 부패 — 스킬이 잘 작동하다 6주 뒤 업스트림 API 변경. 쓰레기 값을 조용히 반환. 사람이 발견할 때까지 방치.
  3. 고아 스킬 — 자율 생성된 스킬의 트리거가 너무 약해서 영원히 매칭 안 됨. 인덱스 토큰만 차지, 실행 0회.

Skillify가 "동사"가 되는 순간

실패 대응을 넘어서 개발 방식 자체가 된다:

  1. 에이전트와 대화하며 문제를 해결
  2. 작동하면 → "skillify it!"
  3. 에이전트가 SKILL.md + 코드 + 테스트 + resolver 자동 생성
  4. 1시간의 시행착오가 영구 인프라가 됨
💡 실제 사례: OAuth webhook 1시간 씨름 → "skillify it" 한 마디 → 다음에 webhook 필요할 때 스킬이 자동 로드
📝 내 생각 쓰기 (수사차록법)
🔁 다시 확인 (반복참정법)
Skillify 10단계 중 3~5단계(테스트 계층)가 각각 잡는 버그 유형의 차이를 설명할 수 있는가?
"오늘만 작동하는 코드"와 "skill"의 차이는 무엇인가?
테스트 없는 스킬 시스템의 3가지 실패 모드를 기억하고 있는가?

Chapter 4. 왜 에이전트 온보딩이 필요한가

🏛️ 축기견초법 築基堅礎

"기초부터 단단히. 전제를 확인하지 않고 구현에 들어가지 않는다."

핵심 개념

HBR(2026-03)의 핵심 주장: "에이전트 도입은 기술 문제가 아니라 일을 관리하는 방식의 문제다." 에이전트를 새로운 소프트웨어로 취급하면 실패한다. 신입 직원처럼 온보딩해야 한다.

WEF도 같은 결론: "에이전트 온보딩은 신입 직원과 같은 수준의 엄격함으로 접근해야 한다."

왜 이게 필요한가?

Microsoft는 e커머스 어시스턴트에서 26개 에이전트를 운영했다. 에이전트가 늘어날수록:

  • 역할이 겹치는 에이전트끼리 충돌
  • 비슷한 이름/설명으로 오케스트레이터가 잘못된 에이전트 선택
  • 사용자 신뢰 하락

온보딩 프로세스를 도입한 후: 2,370개 시나리오에서 recall@5 98%, 최종 invoke 정확도 95%

비교: 온보딩 있는 조직 vs 없는 조직

권한전부 열림최소 권한 → 증명 후 확장
평가"잘 되는 것 같아"정기 평가 + 메트릭

실패 대응프롬프트 수정스킬 생성 + 테스트

Chapter 5. Agent Card — 에이전트의 이력서

🏛️ 선정문목법 先定門目

"목차와 구조를 먼저 세운다. 바로 쓰기 시작하지 않는다."

핵심 개념

WEF는 Agent Card 개념을 제시했다 — 에이전트의 "이력서". 7개 차원으로 에이전트의 행동적·운영적 프로필을 정의한다:

차원질문예시
Function이 에이전트는 무슨 일을 하는가?캘린더 검색, 코드 리뷰
Role전문가인가, 범용인가?특화 vs 제너럴리스트
Predictability결정론적인가, 비결정론적인가?grep = 결정론, 요약 = 비결정론
Autonomy얼마나 독립적으로 판단하는가?연구만 vs 실행+보고
Authority어떤 권한/접근이 허용되나?읽기전용 vs 쓰기 가능
Use case어떤 도메인/워크플로우에 배치되나?고객 지원, 개발 보조
Environment운영 환경이 단순한가 복잡한가?단일 시스템 vs 멀티 플랫폼

핵심 구분: 자율성 ≠ 자동화

WEF의 중요한 관찰: 자율성(Autonomy)은 "무엇을 할지 선택하는 유연성"이고, 자동화(Automation)는 "정해진 것을 실행하는 신뢰성"이다. 이 둘은 시스템 고유 속성이 아니라 설계 선택이다.

💡 로봇 청소기: 낮은 권한, 중간 자율성, 단순 환경. 개인 비서: 높은 권한, 높은 자율성, 복잡 환경 → 비서에게 훨씬 더 엄격한 감독 필요.

HBR의 Job Description

Agent Card를 더 실용적으로 확장하면 HBR의 에이전트 Job Description이 된다:

  • 이 에이전트의 책임은 무엇인가?
  • 이 에이전트의 책임이 아닌 것은 무엇인가?
  • 결정 경계는 어디인가?
  • 어떤 권한을 가지는가?
  • 언제 인간의 승인을 구해야 하는가?

Chapter 6. 권한 설계와 단계적 승격

🏛️ 정과실천법 定課實踐

"목표량을 먼저 정하고 그대로 실천한다."

핵심 개념

Bill Eisenhauer(LinkedIn)의 핵심 주장: "에이전트 유형을 먼저 분류하고, 그다음에 설정하라." 모든 에이전트가 같은 권한으로 시작해서는 안 된다.

에이전트 권한 레벨

레벨설명예시
Research only조사만. 아무것도 변경 안 함미팅 준비 에이전트
Draft & Recommend초안 작성 + 추천. 실행은 인간콘텐츠 에이전트
Recommend판단하고 추천. 실행은 승인 후고객 지원 라우터
Act & Report실행하고 보고일정 관리 에이전트
Act silently보고 없이 자동 실행로그 정리, 캐시 관리

HBR의 "인턴 원칙"

"인턴으로 고용, 증명하면 정규직." 새 에이전트는 영구 워크플로우에 넣기 전에 성과를 입증해야 한다. 교육 시 주장이나 미래 약속이 아니라 실제 결과로만 판단한다.

승격 게이트 — 구체적 기준

에이전트 유형승격 조건
콘텐츠연속 3개 게시물 수정 없이 채택
고객 지원 라우터50건 이상, 2주간 90%+ 분류 정확도
일정 관리2주간 충돌/누락 제로
리서치80%+ 신호 대 잡음 비율

감독자 시간 투자 곡선

시기감독 수준
Week 1무거운 리뷰 (거의 전부 확인)
Week 3~4첫 승격
Week 6~8두 번째 승격
Week 10~12대시보드 + 예외 관리

약 12주 동안 40시간 투자 → 주당 10~20시간 회수. 이것이 에이전트 온보딩의 ROI.

Chapter 7. 플릿 레벨 검증

🏛️ 공심공안법 公心公眼

"선입견을 배제하고, 내가 이미 알고 있다고 생각하는 것을 다시 의심하라."

핵심 개념

Microsoft의 가장 중요한 발견: 새 에이전트 하나가 전체 플릿에 미치는 영향을 테스트해야 한다. 개별 에이전트가 혼자 잘 작동해도, 기존 에이전트와 겹치면 전체 시스템이 무너진다.

Microsoft의 온보딩 시퀀스

  1. 유일성 체크 — 이름, 설명, 기능이 기존 에이전트와 겹치지 않는지
  2. Utterance 평가 — 샘플 발화가 기존 에이전트의 트리거와 충돌하지 않는지
  3. 임시 시맨틱 캐시 통합 — 새 에이전트를 캐시에 추가하고 전체 테스트
  4. 시스템 전체 응답 평가 — 기존 에이전트의 성능이 저하되지 않았는지
  5. 골든 데이터셋 업데이트 — 새 테스트 케이스 추가

WEF의 멀티에이전트 생태계 리스크

리스크설명
Orchestration drift공유 맥락 없이 에이전트를 연결하면 워크플로우가 취약해짐
Semantic misalignment두 에이전트가 같은 지시를 다르게 해석
Security gaps공유 신뢰 프레임워크 없이 데이터 노출
Cascading effects밀접하게 연결된 에이전트 장애가 전파
Systemic complexity에이전트 수 증가 → 예측 불가 행동 증가

Skillify Step 8과의 연결

Garry Tan의 check-resolvable은 정확히 이 문제를 해결한다. 40개 스킬 중 6개(15%)가 resolver에 등록되지 않은 "dark skill"이었다. 플릿 레벨에서 도달 가능성을 검증하지 않으면, 시스템 역량의 15%가 사라진다.

Chapter 8. SKILL.md + 결정론적 코드

🏛️ 본의본령법 本意本領

"핵심을 건드려 전체를 움직여라. 부가 설명보다 핵심 먼저."

Step 1: SKILL.md — 계약서

Skill은 마크다운 절차서다. 모델에게 "무엇을 하라"가 아니라 "어떻게 접근하라"를 가르친다. 메서드 호출처럼: 같은 절차, 입력에 따라 완전히 다른 출력.

name: calendar-recall
description: "Brain-first historical calendar lookup.
  ALWAYS use this before any live API for any event
  not in the future or the last 48 hours."

안에 들어가는 하드 규칙:

Live calendar APIs are ONLY for events in the FUTURE
or the LAST 48 HOURS.
Everything historical goes through the local knowledge
base first.

Step 2: 결정론적 코드

스킬 파일이 에이전트에게 "이건 결정론적 작업이야"라고 알려주면, 에이전트 자체가 스크립트를 작성한다.

$ node scripts/calendar-recall.mjs search "Singapore"

Found 2 matching day(s):
── 2016-05-07 ──
  Flight to Singapore, Mandarin Oriental check-in
── 2016-05-08 ──
  Lunch with investors at Fullerton Hotel

실행 시간: 100ms 이하. LLM 호출 0. 네트워크 0. 순수 로컬 파일.

🛠️ 실습 (정과실천법)

내 에이전트가 반복적으로 하는 작업 중 하나를 골라 SKILL.md 초안을 써보자:

🔁 다시 확인 (반복참정법)
SKILL.md가 "무엇을"이 아니라 "어떻게"를 가르친다는 것의 의미는?
결정론적 코드를 에이전트가 직접 작성하는 구조의 장점은?
calendar-recall의 "하드 규칙"이 차단하는 구체적 실패 경로는?

Chapter 9. 테스트 3계층

🏛️ 무징불신법 無懲不信

"확인하지 않은 것은 말하지 않는다. 검증 없이 신뢰하지 않는다."

Step 3: Unit Tests

vitest로 순수 함수 테스트. 픽스처 데이터(합성 파일), known input → known output.

잡는 버그: parseEventLine이 유니코드 문자가 포함된 장소를 조용히 드롭. dateFromPath가 윤년 날짜에서 null 반환. formatJson이 참석자 1명일 때 배열 생략.

작지만 치명적. 스크립트 출력이 틀리면 → 스킬이 틀린 답을 만들고 → 에이전트가 자신 있게 틀린 걸 말한다.

Step 4: Integration Tests

실제 데이터와 실제 엔드포인트를 때린다. 픽스처에서는 발견 못 하는 버그를 잡는다:

  • 실제 캘린더 파일의 Windows 줄바꿈
  • 자정을 넘기는 이벤트
  • 시간대 필드가 빠진 항목
  • 캐시가 오래되거나 없을 때의 동작

💡 규칙: "스크립트가 실제 데이터에서 제대로 작동하는지 수동으로 확인하게 된다면, 그 확인 자체가 integration test여야 한다."

Step 5: LLM Evals

LLM-as-judge: 모델이 다른 모델의 출력을 루브릭으로 평가.

context-now용 eval 예시 (35개 daily):

  • "비행기 45분 후 출발하는데 SFO 도착할 수 있을까?" → 에이전트가 context-now.mjs를 실행했는지 체크. 머릿속 계산하면 실패.
  • UTC 타임스탬프 변환 질문 → 스크립트 실행 후 결과 인용이 정답. 멘탈 계산은 설사 정답이어도 실패.

핵심: 틀린 답만 잡는 게 아니라 틀린 프로세스도 잡는다. 이번에 우연히 맞아도 다음엔 틀린다.

Garry의 eval 발견법: "대화 기록에서 'fucking shit'이나 'wtf'를 검색하라. 그게 누락된 테스트 케이스다."

📝 내 생각 쓰기 (수사차록법)
🔁 다시 확인 (반복참정법)
Unit test와 Integration test가 잡는 버그 유형의 근본적 차이는?
LLM eval이 "정답이어도 실패"로 판정하는 경우는 언제인가?
Garry의 eval 발견법("wtf 검색")이 시사하는 것은 무엇인가?

Chapter 10. Resolver와 라우팅

🏛️ 휘분류취법 彙分類聚

"모아서 나누고 분류하여 모아라. 산발된 정보를 체계화하라."

Step 6: Resolver Trigger

Resolver = 맥락의 라우팅 테이블. 작업 유형 X가 나타나면 스킬 Y를 로드한다. 각 스킬은 AGENTS.md에 트리거 엔트리가 있어야 한다.

이 단계가 빠지면: 스킬이 존재하고, 역량이 존재하지만, 시스템이 접근할 수 없다. 병원에 외과의가 있는데 디렉토리에 등록 안 된 것과 같다.

Step 7: Resolver Eval

트리거가 "이 문구는 이 스킬로 라우팅돼야 한다"고 선언한다. Resolver eval은 실제로 그렇게 되는지 테스트한다.

{ intent: 'what time is my meeting',   expectedSkill: 'context-now' }
{ intent: 'find my 2016 trip',          expectedSkill: 'calendar-recall' }
{ intent: 'who is Pedro Franceschi',    expectedSkill: 'brain-ops' }
{ intent: 'save this article',          expectedSkill: 'idea-ingest' }

두 가지 실패 모드:

  • False negative: 스킬이 발동해야 하는데 안 함 → 트리거 설명이 잘못됨
  • False positive: 잘못된 스킬이 발동 → 두 트리거가 겹침

"내 캘린더 내일 뭐 있어?" — calendar-check, calendar-recall, google-calendar 세 스킬 중 어디로? 세 개가 다른 시간 도메인을 담당하는데, 이 문구는 세 개 다 매칭 가능. Resolver eval이 이 모호성을 사용자가 맞닥뜨리기 전에 잡는다.

🛠️ 실습 (정과실천법)

내 에이전트의 스킬 3개를 골라 resolver 매핑 테스트 케이스를 써보자:

🔁 다시 확인 (반복참정법)
Resolver trigger(Step 6)와 Resolver eval(Step 7)이 각각 잡는 문제가 다른데, 그 차이는?
False positive와 False negative 중 어떤 것이 더 위험하고 왜인가?
Resolver eval의 두 계층(구조적 테스트 + LLM 라우팅 테스트)이 모두 필요한 이유는?

Chapter 11. DRY 감사 + E2E Smoke Test + Brain Filing

🏛️ 종핵파즐법 綜核爬櫛

"종합하여 핵심을 찾고, 꼼꼼히 빗질하듯 정렬하라."

Step 8: Check-resolvable + DRY Audit

check-resolvable이 확인하는 3가지:

  1. SKILL.md가 있는 모든 스킬 디렉토리 → resolver에 엔트리가 있는가
  2. 스킬이 참조하는 모든 스크립트 → 실제로 호출 가능한가 (파일 존재, 올바른 export)
  3. 두 스킬의 트리거 설명이 겹쳐서 라우팅 모호성을 만들지 않는가

DRY audit 예시 — 캘린더 도메인 4개 스킬의 레인 분리:

스킬담당 시간 도메인접근 방식
calendar-recall48시간 이전 과거로컬 KB grep
calendar-check오늘~이번 주라이브 API
google-calendar미래 이벤트 CRUDGoogle API
context-now지금 이 순간로컬 스크립트

4개 스킬, 영역 겹침 없음, 각자 레인. 5번째 캘린더 스킬이 다른 레인을 침범하면 DRY audit가 배포 전 실패시킨다.

Step 9: E2E Smoke Test

전체 파이프라인을 end-to-end로 테스트. 모든 조각이 맞아도 연결이 안 되면 실패할 수 있기 때문이다.

  • "싱가포르 언제 갔어?" → calendar-recall.mjs 실행 → 올바른 답 → 올바른 포맷?
  • "다음 미팅 몇 시야?" → context-now.mjs 실행 → 멘탈 계산 안 했는지?

스킬이 맞고, 스크립트가 맞고, resolver가 맞아도 — 에이전트가 전부 무시하고 "즉흥"할 수 있다. Smoke test가 그걸 잡는다.

Step 10: Brain Filing Rules

KB에 쓰는 스킬은 어디에 저장할지 규칙이 필요하다. 사람은 people/, 회사는 companies/, 정책은 civic/에.

13개 brain-writing 스킬 중 10개가 잘못된 디렉토리에 저장하고 있었다. 각자 경로를 하드코딩했기 때문. filing rules 문서를 만들고 스킬이 저장 전에 참조하게 하니 오분류 제로.

🔁 다시 확인 (반복참정법)
check-resolvable의 "dark skill" 문제를 한 문장으로 설명할 수 있는가?
E2E smoke test가 다른 테스트와 다른 점은 무엇인가?
Brain filing rules가 없을 때 발생하는 구체적 실패 모드는?
